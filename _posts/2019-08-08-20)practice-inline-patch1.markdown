* 인라인 패치란? 실행 압축되거나 암호화된 파일을 패치할 때 자주 사용되는 기법이다.  

## 20.1) 인라인 패치  
* 원하는 코드를 직접 수정하기 어려울 때 간단히 **코드 케이브(Code Cave)**라고 하는 패치 코드를 삽입한 후 실행해 프로그램을 패치시키는 기법이다.  
* 대상 프로그램이 실행 압축(혹은 암호화)되어 있어서 파일을 직접 수정하기 어려운 경우 많이 사용되는 기법이다.  
* 코드 케이브 내에 패치 코드를 실행 후(이미 OEP는 복호화되었기 때문에 그대로 수정 가능) OEP로 가면 된다. -> **실행할 때마다(별도의 패치 코드를 통해) 매번 프로세스 메모리의 코드를 패치한다.**  

| |코드 패치|인라인 패치|  
|:--------|:--------------|:------------|  
|대상|파일|파일 & 메모리|  
|횟수|1번|파일에는 1번만, 메모리는 실행될 때마다|  
|방법| Direct(원하는 위치에 직접 패치)|Indirect(코드 케이브를 미리 설치한 후 메모리에서 원하는 영역이 복호화되었을 때 패치)|

## 20.3) 디버깅 - 코드 흐름 살펴보기  
![decolop1](/asset/decolop1.JPG)  

* XOR BYTE PTR DS:[EBX], 44 명령으로 특정 영역(004010F5 ~ 00401248, ECX 값에서 추정했음)을 XOR 명령으로 복호화한다.  

![4010](/asset/4010.JPG)  

* XOR BYTE PTR DF:[EBX], 7 명령으로 특정 영역(00401007 ~ 00401085, ECX 값에서 추정했음)을 XOR 명령으로 복호화 한다.  
* 그 후, 004010DB 주소의 XOR 명령에 의해서 4010F5 ~ 401248 영역이 복호화된다. -> 이 영역이 이중으로 암호화 되어있음을 알 수 있다.  

![401039](/asset/401039.JPG)  

* 따라가다 보이는 401039 함수에서 눈여겨 볼 점은 401046주소에 있는 **Checksum 계산 루프**이다.  
* 먼저 401041 주소의 MOV EDX,0으로 EDX를 초기화 시키고, 401046의 ADD 명령으로 특정 주소 영역(4010F5 ~ 401248)에서 4바이트 단위로 순차적으로 값을 읽어들여서 EDX 레지스터에 덧셈 연산으로 누적시킨다. -> **누적된 값이 Checksum** 값이다.  
* 이 Checksum 계산 영역이 위에서 봤던 이중으로 암호화되어 있는 영역이다. -> 이 영역에 패치하려는 문자열이 존재할 것으로 예상된다.  
* 401062 ~ 401068 주소의 CMP/JE 명령은 이렇게 계산한 Checksum(EDX 레지스터에 저장된) 값이 31EB8DB0과 같다면(코드가 변조되지 않았다면), 401083 주소의 JMP 명령에 의해 OEP(40121E)로 간다. 하지만 Checksum 값이 다르다면 에러메시지를 출력하고 프로그램이 종료된다.   

![oep1](/asset/oep1.JPG)  
* OEP 코드로 들어간 모습이다. 내용은 다이얼로그를 실행시키는 것이고, 40123E 주소의 CALL user32.DialogBoxParamA() 명령을 실행하면 다이얼로그가 나타난다.  

```  
DialogBoxParam() API 정의  
INT_PTR WINAPI DialogBoxParam(  
    __in_opt HINSTANCE hInstance,  
    __in     LPCTSTR lpTemplateName,  
    __in_opt HWND hWndParent,  
    __in_opt DLGPROC lpDialogFunc,  
    __in     LPARAM dwInitParam  
);  
```  

* DialogBoxParam() API의 4번째 lpDialogFunc 파라미터가 Dialog Box Procedure 주소이다.(OllyDbg 에서는 DlgProc으로만 표시함.) 위의 그림에서 40122C 주소에 DlgProc 주소는 4010F5라는 것을 알 수 있다.  

![4010f5](/asset/4010f5.JPG)  

* 여기서 40110A ~ 401133, 4011A5, 4011BB를 패치해야한다.  
* 이 프로그램은 문자열이 이중으로 암호화 되어있을 뿐만 아니라 Checksum값도 확인하면서 변조 여부를 판단하기 때문에 일반 패치는 힘들다. -> 인라인 패치가 필요하다!  

## 20.4) 코드 구조  

```  
401000 [EP Code]  
40109B  [Decoding Code]  
4010A3  XOR (4010F5 ~ ), 44  
4010C8  XOR (401007 ~ 401083(401039 함수)), 7  
4010DB  XOR (4010F5 ~ ), 11  
401039  401039 함수  
401046      Checksum 4010F5~  
401090      XOR (40124A ~), 17  
401083      JMP OEP  
```  

## 20.5) 인라인 패치 실습  

```  
1. 파일의 적절한 위치에 문자열을 패치시키는 코드를 삽입한다.(코드 케이브)  
2. 401007 ~ 401083 영역의 JMP OEP 명령을 JMP Patch Code로 수정한다.(그 부분이 암호화 되어있음은 고려해야한다.)  
3. 코드 케이브에서 OEP로 JMP하면 인라인 패치 완성.  
```  

# 20.5.1) 패치 코드의 위치는?  
1. 파일의 빈 영역에 설치  
2. 마지막 섹션을 확장한 후 설치  
3. 새로운 섹션을 추가한 후 설치  

* 보통 패치 코드의 크기가 작은 경우 1번, 나머지 경우는 2, 3을 이용한다.  

![text](/asset/text.JPG)  

* .text 섹션을 살펴보면, Virtual Size = 280인데, Size of Raw data = 400 이다. -> 파일에서의 크기는 400이지만 메모리에 로딩시 280이 된다는 이야기 이므로, Pointer to Raw Data = 400 에 따라 680 ~ 800 은 비어 있다는 의미가 된다.  
![empty](/asset/empty.JPG)  

# 20.5.2) 패치 코드 만들기  
* OllyDbg 에서 OEP(40121E)까지 간다.  
* File Offset 680 ~ 800은 VA로 변환하면 401280 ~ 401400(식대로 계산 해볼 것) 이므로 OllyDbg에서 책에 나오는대로 패치하면 밑의 사진처럼 된다.  
![patch](/asset/patch.JPG)  

* 40128F, 4012A0의 REP MOVSB 명령에 의해서 원래 문자열들이 패치된다.  

# 20.5.3) 패치 코드 실행하기  
* 401083 주소의 JMP OEP 명령어를 코드 케이브로 변경하면 된다.  
* 이때 유의할것은 이 영역(401083)은 원래 암호화되어 있는 영역이라는 것이다. 이 영역은 XOR 7으로 암호화 된 영역이므로 JMP 코드 케이브를 쓸때(JMP unpackme.00401280)을 E9 F8 01 이라 그대로 쓰면 안된다.  

```  
E9 XOR 7 = EE  
F8 XOR 7 = FF  
01 XOR 7 = 06  
```

## 마무리  
* 어셈블리어 코드를 내 손으로 작성한 것은 아니므로 코드에 대한 공부는 좀 더 필요하다.
